# PORTFOLIO.md

## Why This Project Exists

**Version:** 0.1

This document exists to protect the project from becoming a well-designed movie application.

Pentimento is a portfolio project.

Every design and engineering decision should strengthen the portfolio before it strengthens the product.

The project is successful if it becomes memorable.

Not if it becomes feature complete.

---

# The Portfolio Goal

Pentimento should demonstrate a capability that none of my other projects demonstrate.

The portfolio should feel like a collection of different investigations rather than variations of the same project.

---

# What This Project Must Prove

Invisible Interfaces proves:

I can investigate interaction.

---

Interaction Atlas proves:

I can build systems for interaction design.

---

Pentimento must prove:

I can identify an overlooked asymmetry between people and software, take a position on it, construct an original interaction grammar around it, and build the working proof — a system that narrates a person and structurally submits to their corrections.

The position, in one line: software that narrates a person owes that person a right of reply.

If the project does not demonstrate this capability — and this position — it has failed.

---

# The Ideal Reaction

The goal is not for reviewers to say:

"This UI looks beautiful."

or

"Nice animations."

The goal is:

"I've never thought about movie history this way."

or

"That's an interesting way to think about cultural behavior."

or

"I wish this existed."

The project should be remembered for its idea.

The visuals should reinforce the idea.

Never replace it.

---

# The Five Portfolio Questions

Every feature must answer YES to all five.

---

## Question 1

Does this demonstrate a capability that my portfolio does not already demonstrate?

If no:

Delete it.

---

## Question 2

Would this still be interesting without beautiful visuals?

If no:

The idea is weak.

Improve the idea.

---

## Question 3

Would someone remember this interaction six months later?

If no:

It is not distinctive enough.

---

## Question 4

Does this strengthen the research contribution?

If no:

It probably belongs in another product.

---

## Question 5

Would a real user naturally understand why this exists?

If no:

Simplify.

---

# The CMU Test

Imagine presenting this project during a graduate portfolio review.

Every screen should answer one of these questions.

What behavior are you investigating?

What evidence supports your theory?

What did you discover?

Why couldn't an existing product solve this?

What new interaction model did you propose?

If a screen cannot answer one of these questions, reconsider its purpose.

---

# The Recruiter Test

Imagine a recruiter spends four minutes on the project.

What should they remember?

Not the color palette.

Not the typography.

Not the stack.

They should remember exactly three things.

1.

This person investigates problems before designing solutions.

2.

This person creates original interaction models.

3.

This person can transform abstract research into polished products.

Everything else is secondary.

---

# The Originality Test

Ask before every implementation.

Could this exist inside Letterboxd?

If yes:

The idea is probably too small.

Could Netflix build this as a recommendation feature?

If yes:

The contribution is probably technological rather than conceptual.

Could someone describe this as "AI for movies"?

If yes:

The project has lost its identity.

Originality comes from the interaction model.

Not the implementation.

---

# Signature Moments

The project should contain a small number of unforgettable moments.

Not dozens of good ones.

A reviewer should pause.

Think.

Interact.

Understand.

Move on.

Quality of moments is more important than quantity.

---

# The One-Minute Rule

A reviewer should understand the project's central idea within one minute.

If understanding requires reading long paragraphs, redesign the experience.

The interface should teach the thesis.

---

# The Ten-Minute Rule

A reviewer who spends ten minutes exploring should discover increasing depth.

Surface understanding should become deeper understanding.

The project should reward curiosity.

---

# The Case Study Rule

The product and the case study should tell the same story.

Neither should compensate for weaknesses in the other.

A weak product cannot be rescued by a strong case study.

A weak case study cannot explain a confusing product.

---

# Build Less

When deciding between:

Three good interactions

or

One unforgettable interaction

Always choose one unforgettable interaction.

This project is remembered through moments.

Not completeness.

---

# Remove Anything Generic

Delete features that exist because every app has them.

Examples.

Followers.

Likes.

Notifications.

Achievements.

Profiles with statistics.

Infinite feeds.

Complex settings.

These features add familiarity.

Not originality.

---

# Build Only What Supports The Thesis

Every screen should answer:

How does this help users recognize meaningful change in themselves?

If the answer is unclear, remove the screen.

---

# Creative Risk

This project should take at least one risk that would probably never survive inside a commercial product.

Portfolio projects exist to explore.

Not to maximize market adoption.

Choose memorable over safe.

Thought-provoking over efficient.

Original over familiar.

---

# The Final Question

At the end of every implementation session, ask:

If someone deleted every movie poster from this project...

Would the underlying interaction still be interesting?

If the answer is no...

The project is relying on aesthetics rather than ideas.

Keep working.

---

# Definition of Success

Pentimento succeeds when people describe it as:

"A new way of understanding cultural evolution."

Not:

"A beautifully designed movie app."

That single sentence should guide every decision throughout the project.
